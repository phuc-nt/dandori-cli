# 2026-05-04 — Phase 05 / Step 04 — Audit chain external anchor

## What shipped

**The audit_log hash chain now has an external witness.** Before today, `dandori audit verify` only proved the local chain was internally consistent — anyone with write access to the SQLite DB could rebuild a fresh, valid-looking chain and the verifier would happily green-tick it. The new `audit_anchors` table records periodic snapshots of the chain's tip; an optional Confluence page is the off-host witness; and `dandori audit verify --with-anchor` cross-checks them both.

If the chain is rebuilt or rewritten, the recorded `(last_audit_id, last_curr_hash)` no longer reproduces, and verification flips to `INVALID` with a "anchor mismatch" reason. The Confluence row is the human-auditable copy of the same evidence.

### Backend

- `internal/db/v10_to_v11_migration.go` — new. `MigrationV10ToV11` SQL: creates `audit_anchors` (`id`, `anchored_at`, `last_audit_id` UNIQUE, `last_curr_hash`, `confluence_page_id`, `confluence_version`, `status`), plus 2 indexes (`anchored_at`, `last_audit_id`). UNIQUE makes re-anchoring the same tip a silent no-op.
- `internal/db/schema.go` — `SchemaVersion = 11`. DDL appended to canonical `SchemaSQL` so fresh installs land at v11 directly.
- `internal/db/migrate.go` — v10→v11 branch added.
- `internal/db/audit_anchors.go` — new. Helpers: `LatestAuditTip()` (id+hash of newest audit_log row, returns 0/"" on empty log — not an error), `LatestAuditAnchor()`, `ListAuditAnchors(limit)`, `InsertAuditAnchor(...)` using `INSERT OR IGNORE` for idempotency.
- `internal/db/audit_queries.go` — new `VerifyAuditChainWithAnchors(limit)` runs the existing chain check first; if internally valid, walks every recorded anchor and asserts the audit_log row at `last_audit_id` exists and its current `curr_hash` still matches what was anchored. Drift → `Valid=false`, `Reason` prefixed with `anchor mismatch:`. Internal check failure short-circuits — no point cross-checking a chain that doesn't even self-verify.
- `internal/server/audit_endpoints.go` — `/api/audit-log/verify` now honors `?with_anchor=true` (alongside the existing `?limit` and `?full=true`).

### Confluence witness

- `internal/confluence/audit_anchor.go` — new. `UpsertAuditAnchorPage(client, spaceKey, title, row)` finds-or-creates a witness page (default title `"Dandori Audit Anchors"`) and appends one HTML-storage table row per anchor. Idempotent on `last_audit_id`: if the row already exists on the page, the call short-circuits and returns the unmodified page. Defines a small `AuditAnchorClient` interface (subset of `*confluence.Client`) so tests inject a fake without spinning up Confluence.

### CLI

- `cmd/audit.go` — new. Top-level `dandori audit` group with two subcommands:
  - `dandori audit anchor [--local-only] [--title T] [--space K]` — reads the current tip via `LatestAuditTip`, idempotently writes a row to `audit_anchors`, and (when Confluence is configured + `--local-only` not set) upserts a row on the witness page. Reports the local anchor #, last_audit_id, short hash, and status.
  - `dandori audit verify [--with-anchor] [--limit N]` — runs the existing internal verifier, optionally chained with anchor cross-checks. Exit code is non-zero on `Valid=false`.
- Confluence client construction reuses the same fallback rules as `conf-write` (dedicated `confluence.user/token`, falling back to Jira creds for single-tenant Cloud setups).

### Demo seed

- `internal/demo/seed.go::SeedCrossProject` — appends 4 deterministic `audit_log` entries (task.create, task.assign, policy.update, verification.pass) so demo walkthroughs have a real chain to anchor and verify.
- `ResetDB` extended to wipe `audit_anchors` and `audit_log` so `--reset` always lands in a clean, anchorable state.

## Live verification

```
$ /tmp/dandori-p4 demo --reset --seed --use --variant cross-project
$ /tmp/dandori-p4 audit verify
✓ chain valid (4 entries)
$ /tmp/dandori-p4 audit anchor --local-only
Local anchor #3: last_audit_id=15 hash=27793ddd1b1c… status=local-only
$ /tmp/dandori-p4 audit verify --with-anchor
✓ chain valid (4 entries) · 1 anchors cross-checked

# Re-anchor with no new rows: silent no-op
$ /tmp/dandori-p4 audit anchor --local-only
Already anchored at id=15 (anchor #3, …) — no-op

# Rebuild-attack simulation: wipe most of audit_log so the anchored tip vanishes
$ sqlite3 ~/.dandori/demo.db "DELETE FROM audit_log WHERE id > 1;"
$ /tmp/dandori-p4 audit verify
✓ chain valid (0 entries)        # internal-only check is fooled
$ /tmp/dandori-p4 audit verify --with-anchor
Error: ✗ chain INVALID at id=15 (idx 0): anchor mismatch: audit_log row 15 (anchored at …) is missing
```

The final two commands are the proof point: a chain rewrite that internal verification accepts is caught by the anchor witness.

The Confluence path was exercised via unit tests (no Cloud creds in this dogfooding setup); CLI behavior with real creds will be validated when QA runs `dandori audit anchor` against the staging Confluence space.

## Notes / debt left

- The witness page format is HTML storage-format only (no ADF). The rest of the package writes storage too, so this stays consistent — but it does mean Cloud's modern editor will render the table read-only without rich formatting. Fine for an audit witness.
- No daily scheduler ships in this phase. Operators run `dandori audit anchor` manually or via cron. A scheduled push is a natural Phase 06 follow-up — the surface (idempotent CLI + Confluence upsert) is already designed for it.
- `audit_anchors` is wiped by `dandori demo reset` so demos always start clean. In a real install nothing wipes it — that's intentional, anchors are tamper-evidence and should outlive demo cycles.
- Bug-type detection in the existing buglinks hook still uses substring `bug` (Phase 02). Out of scope here.
