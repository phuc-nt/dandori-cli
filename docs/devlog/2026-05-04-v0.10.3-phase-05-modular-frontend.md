# 2026-05-04 — Dashboard v2 Phase 05 / Step 03 — Modular frontend (QA + Audit)

## What shipped

**Started the long-overdue split of `cmd/web/dashboard/app.js` (2846 lines → 2552) into ES6 modules.** This step extracts the QA view (6 renderers) and the Audit view (3 entry points) — the two most recently added persona blocks, well-isolated from the rest of the file. The remaining 2552 lines (PO/Tasks/Engineering/Admin views, URL state machine, sprint picker, KPIs, route handlers) stay as a single classic script for now and get split in a later cycle.

### Why partial, not full

`app.js` is a 2846-line file of top-level functions referenced by ~30 inline `onclick="..."` attributes in `index.html`. A full rewrite into modules in one PR is high-risk (every onclick becomes a window-global rebind, plus the URL state machine and persona switcher are tightly coupled). Splitting QA + Audit first proves the module layout, the static-asset pipeline, and the bootstrap pattern with low blast radius.

### Frontend

- `cmd/web/dashboard/widgets/shared.js` — new. Helper-only (no DOM, no fetches): `chartColors`, `escapeHtml`, `truncate`, `heatBucket`, `shortRepo`. Mirrors the closure-scoped helpers still present in `app.js` until the full split lands.
- `cmd/web/dashboard/widgets/qa.js` — new. 6 renderers (`renderQATimeline`, `renderQAScatter`, `renderQACommitMsg`, `renderQABugHotspots`, `renderQARework`, `renderQAIntervention`) + `loadQAView`. Reads Chart.js off `window.Chart` (UMD `<script>` in `index.html`). Module-level chart cache so repeat renders `destroy()` the previous instance before re-instantiating.
- `cmd/web/dashboard/widgets/audit.js` — new. `loadAuditView`, `reloadEventStream`, `loadAuditLog`, `verifyAuditChain`.
- `cmd/web/dashboard/app.js` — QA + Audit functions removed (lines 2545–2846 deleted, 294 lines net). Persona-switch and `loadAll()` callers swapped to optional-chain form: `window.loadQAView?.()` / `window.loadAuditView?.()` so they no-op cleanly if the module hasn't installed its globals yet.
- `cmd/web/dashboard/index.html` — added a bootstrap `<script type="module">` block at end of `<body>`, after the existing `<script src="/static/app.js">`. It imports the two view modules and assigns `loadQAView`, `loadAuditView`, `reloadEventStream`, `verifyAuditChain` to `window`. Inline `onclick="reloadEventStream()"` and `onclick="verifyAuditChain()"` keep working unchanged. The bootstrap also re-triggers the active persona's loader once globals are installed — module scripts are always deferred, so app.js's initial `loadAll()` may have raced past the QA/Audit calls (now guarded by `?.()`); the bootstrap fills that gap.

### Server

No changes. `//go:embed all:web/dashboard` (the `all:` prefix means recursive) automatically picked up the new `widgets/` subdirectory, and `http.FileServer` auto-serves any path under `/static/`. Verified: `/static/widgets/{shared,qa,audit}.js` all return `200`.

## Notes / debt left

- `app.js` still has closure-scoped helpers (`escapeHtml`, etc.) duplicated with `widgets/shared.js`. Kept in lockstep until the rest of `app.js` migrates — collapsing that duplication earlier would force a full rewrite of every PO/Engineering/Admin renderer in this PR.
- The bootstrap module's "re-trigger active persona" is a workaround for the deferred-module / classic-script load-order race. It goes away once the entire app is modules and there's no classic `app.js` loading `loadAll()` first.
- Future cycles split the rest by persona: PO → Engineering → Admin → URL state machine. Each is roughly the same shape as this one (extract module, expose entries on window, optional-chain the callers in app.js).
