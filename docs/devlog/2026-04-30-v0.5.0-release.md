## Why one consolidated v0.5.0 (not v0.5/v0.6/v0.7)

Three internal-only PR cycles between v0.4.0 and now never had release tags cut. CHANGELOG had drafted v0.5.0 (G6) + v0.6.0 (G7) entries, but nothing was published. Consolidating avoids:
- 3 GitHub releases for users to wade through
- Backdated tags at older commits (cleaner git history)
- Confusing semver story when v0.6.0 → v0.7.0 are 1-day apart

Cost: lost per-feature changelog granularity. Mitigated by `### Added — G6` / `### Added — G7` subsections in the v0.5.0 entry.
