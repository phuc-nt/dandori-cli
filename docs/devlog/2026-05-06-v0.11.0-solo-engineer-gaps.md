# 2026-05-06 — v0.11.0 release (solo-engineer self-measurement gaps)

## What this tag bundles

v0.11.0 closes 3 self-measurement gaps that a solo engineer running dandori-cli locally couldn't answer with v0.10.6. The audit (`plans/reports/explore-260506-0759-solo-engineer-toolset-audit.md`) identified 5 gaps total; v0.11 ships the 3 that are pure-solo (don't need Jira "Done" config). The other 2 (cost-per-merged-task, scheduled snapshots) deferred to v0.12.

## What shipped

| Gap | Question answered | CLI | Endpoint | Dashboard |
|-----|-------------------|-----|----------|-----------|
| Agent × Task affinity | Which model for which task type? | `dandori analytics agents --by-task-type` | `/api/analytics/agent-task-affinity` | Engineering → Agents heatmap |
| Structured RCA | Where to focus fixes? | `dandori analytics rca --since 28d` | `/api/rca/breakdown` | QA rework doughnut tooltip |
| Trend analytics | Am I getting better? | `dandori analytics trend --metric M --window 7d` | `/api/trends/{success-rate,cost-per-run,rework-rate}` | Engineering → Cache & Cost line chart |

3 commits, one per phase:
- `d885e59` Phase 01 — Agent × Task affinity matrix
- `6c5d541` Phase 02 — Structured RCA aggregation
- `c859859` Phase 03 — Week-over-week trend analytics

## Architecture insights

- **No schema migration** — the original audit assumed `runs.task_type` existed. It doesn't. Phase 01 derives task type at query time from `runs.jira_issue_key` (Jira issuetype lookup → prefix-parse fallback → `(unknown)`). All-local, all-derived. Same insight applied to Phase 02 (correlations done via in-memory aggregation over existing event stream).
- **Existing rework cause enum was complete** — `internal/db/run_outcome_reason.go` already had stable codes from v0.10.1. Phase 02 only aggregates; it doesn't extend the enum. `scope_creep` deferred to v0.12.
- **`kpi_strip.go` is daily-only** — couldn't retrofit week-window queries onto it without regression risk. Phase 03 introduces a separate `internal/db/trends.go` with its own gap-fill helper. Cleaner separation; no behavior change to existing KPI strip widget.

## Surprises

1. SQLite `strftime('%Y-%W')` uses Sunday-start weeks while Go's `ISOWeek()` uses Monday-start. Cosmetic off-by-one in gap-fill on weeks containing a Sunday/Monday boundary. Tracked for v0.11.1 with a unit test that seeds boundary data.
2. Engineering view doesn't have a dedicated `engineering.js` widget file — the audit hinted otherwise. New widgets created as standalone files (`affinity.js`, `trend.js`) and wired into `app.js`. Followed the same pattern QA + Audit views already use.
3. CLI flag `analyticsSince` is shared between `analytics mix` and the new `analytics agents --by-task-type`. Cobra parses per-subcommand so it's safe at runtime, but minor cleanup candidate.

## Solo-engineer flow now

1. `dandori run -- claude` — capture runs as before
2. `dandori analytics trend --metric success-rate` — see weekly slope ("improving 2pp/week")
3. `dandori analytics rca --since 28d` — see top rework cause + which agent triggers it most
4. `dandori analytics agents --by-task-type` — pick the right model for the next task
5. `dandori dashboard` (loopback default) — same data visually, plus rework doughnut tooltip + Engineering line chart

No Jira config needed for any of the above. No team server. No Postgres. Single SQLite DB, single binary.
