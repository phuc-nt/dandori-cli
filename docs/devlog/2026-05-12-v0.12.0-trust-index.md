# 2026-05-12 — v0.12.0 release (Trust Index + Code Acceptance Rate)

## What this tag bundles

v0.12 closes the "is the agent earning more authority?" gap. v0.11 told a solo engineer how the agent ran (success, cost, rework); v0.12 reads how much of the output got accepted as-shipped and rolls that into a single composite Key Result, the Trust Index, with an autonomy band so the next assignment decision has a defensible anchor.

Two new metrics, one composite, one tile. No schema changes.

## What shipped

| Layer | Where | What changed |
|-------|-------|--------------|
| **Metric framework** | `docs/reference/04-metric-framework.md` | New 12-metric DORA + SPACE + DevEx set with Goodhart-aware design, validation matrix, AI-CFR proxy disclosure |
| **Code Acceptance Rate trend** | `internal/db/trends.go` + `/api/trends/acceptance-rate` + `dandori analytics trend --metric acceptance-rate` | Weekly buckets of `SUM(agent_lines) / SUM(agent + human)` over `task_attribution.jira_done_at`, gap-filled |
| **Trust Index composite** | `internal/db/trust_index.go` + `/api/metrics/trust-index` + `dandori analytics trust` | `0.40·acceptance + 0.35·(1 − ai_cfr) + 0.25·(1 − clamp(intervention, 0..1))` → 0–100 with band classification |
| **Dashboard Trust tile** | `cmd/web/dashboard/widgets/trust.js` + 4-col grid | 4th tile on the Engineering solo-KPI strip with band-coloured chip and component breakdown |

## Why this patch

Live-test of v0.11.2 confirmed the dashboard could answer "how did agent runs go this week?" but not "should the agent get harder tasks?" The metrics for the second question were either uncomputed (Acceptance Rate had no aggregator) or scattered (Intervention sat in one table, attribution in another). Tech lead's framing: *"raw data is there; designing a metric you can set a goal on and defend in a 1:1 is a different problem."*

The framework doc fixes the design half. The Trust Index fixes the surfacing half — one number, one band, three components on demand.

## Architectural choices

- **Composite lives in `internal/db/`, not `internal/analytics/`.** The analytics package is gated by `//go:build server` (Postgres-only). Trust Index runs against LocalDB / SQLite alongside the other trend computations.
- **AI-CFR is the v0.12 interim proxy**, not the DORA "reverted-within-7d" definition. The proxy is `SUM(total_iterations > 1) / COUNT(tasks)` over `task_attribution`. True CFR needs PR/deploy event capture (revert detection) — landing in v0.13. The framework doc §9 marks the proxy as ⚠️ partial so future-me can't confuse it for the real thing.
- **Intervention rate is runs-weighted** and can legitimately exceed 1.0 (avg interventions per run > 1). The `(1 − rate)` term is clamped to `[0, 1]` so a noisy agent cannot drive the composite negative. Tested.
- **`HasData=false` degrades to `band="no-data"`** when either `task_attribution` or `runs` is empty in the window. The frontend renders a grey chip + "Need ≥1 attributed task and ≥1 run" message instead of a misleading 0.
- **4-column grid uses a new ≤ 1200px breakpoint.** Three tiles at 33% each was fine; four tiles at 25% gets cramped under ~1200px, so the strip collapses to 2 columns there and to 1 below 769px.

## What's not in this release

- **True DORA AI-CFR.** Needs PR/deploy/revert event capture. Deferred to v0.13.
- **Trend chart for Trust Index over weeks.** Out of scope for v0.12 — current tile is the 28-day snapshot. The week-over-week trend variant would need the composite recomputed per ISO-week bucket, which is straightforward but not on this tag.
- **Goal-setting UI.** The framework doc proposes a quarterly cadence (set band target → measure → adjust) but the dashboard doesn't yet capture targets. Manual in v0.12.
