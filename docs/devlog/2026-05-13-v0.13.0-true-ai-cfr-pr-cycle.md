# 2026-05-13 — v0.13.0 — True AI-CFR + PR Review Cycle Time

## What shipped

Two metric upgrades that close the v0.12 "interim proxy" gap noted in `docs/reference/04-metric-framework.md` §9:

1. **True AI-CFR** replaces the multi-iteration proxy with real GitHub PR events.
   `AI-CFR = COUNT(DISTINCT PRs reopened_within_7d OR reverted_post_deploy) / merged_PRs`
   Captured via a new pull-based GitHub sync (`dandori sync --github-only`).
   Proxy is retained as automatic fallback when no merged PRs are in the window — Trust Index keeps working on fresh installs.
2. **PR Review Cycle Time** — new diagnostic metric. `p50 / p75` of `(first_approval_at − submitted_at)` hours over PRs merged in the window. Surfaces process-vs-capacity bottlenecks per framework §8.

Both ship under one release because they share the same data pipeline (`pr_events` table) and the same revert/approval detection paths.

## Architecture

**Pull-based sync, no webhook.** Chose REST polling over webhooks because dandori-cli targets solo engineers and small teams — webhook setup is a non-trivial onboarding cost, and the 28d/90d Trust window doesn't need real-time freshness. `dandori sync --github-only` is best-effort: GitHub failures warn but don't fail the rest of the sync.

**Schema v12** — additive `pr_events` + `sync_state` tables (no data migration). Idempotent `UPSERT ON CONFLICT(repo, pr_number)` so repeated syncs are safe.

**Two-pass orchestration** — first pass upserts all PRs and detects reopens; second pass resolves reverts. Necessary because GitHub returns PRs in update-desc order, putting revert PRs (newer) before their originals in the batch — without the split, a revert can land before its target exists.

**Revert detection via title regex** — `^Revert "<original-title>"$`. Matches GitHub's UI default. PRs created via API with custom titles won't match; documented as a known limitation.

**Median in Go, not SQLite** — SQLite has no `MEDIAN()`. Pulled durations into a `[]float64`, sorted, and applied linear interpolation between the two nearest ranks (Datadog / Grafana convention). For ~65 PRs per window this is plenty accurate.

## Decisions worth remembering

- **No shared HTTP helper.** Resisted the urge to factor a `github.Client` abstraction — current usage is one orchestrator + one ListPRs/ListReviews pair. YAGNI per `CLAUDE.md`. Revisit at v0.14 if a second caller appears.
- **Trust weights unchanged.** Only the `ai_cfr` source swaps. Historical scores stay comparable; the only user-visible change is the `proxy` / `github` badge next to CFR.
- **PR Cycle Time is diagnostic, not a KR.** Resisted including it in the Trust composite — it would conflate quality (CFR) with process (review latency). Lives next to Trust on the dashboard but is read independently.
- **`cfr_source` flows through JSON for free.** Server marshals `db.TrustResult` directly; the struct tag does the work — no endpoint changes.

## Migration notes

No data migration. Users upgrading from v0.12 will see `cfr_source = "proxy"` until they run `dandori init` (which now adds the GitHub block) and `dandori sync --github-only`. Once a PR merges within the window, the source flips to `pr_events` automatically.
