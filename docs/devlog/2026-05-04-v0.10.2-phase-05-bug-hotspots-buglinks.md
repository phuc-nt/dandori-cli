# 2026-05-04 — Dashboard v2 Phase 05 / Step 02 — Real bug hotspots via buglinks

## What shipped

**Replaces the noisy regression proxy in `qa_queries.go::BugHotspots` (lint Δ>0 OR tests Δ<0) with a typed `buglinks` table fed by the task-done hook.** The QA Bug Hotspots widget now counts distinct Jira bugs whose offending run lives in each (repo, week) cell instead of any in-flight quality regression.

### Backend

- `internal/db/v9_to_v10_migration.go` — new `MigrationV9ToV10` SQL: creates `buglinks` table (`id`, `jira_bug_key`, `run_id` FK to `runs`, `reason`, `linked_at`, `linked_by`) plus 3 indexes (`run_id`, `linked_at`, `jira_bug_key`). UNIQUE constraint on `(jira_bug_key, run_id)` makes the insert idempotent.
- `internal/db/schema.go` — `SchemaVersion = 10`. Buglinks DDL appended to canonical `SchemaSQL` so fresh installs land at v10 directly.
- `internal/db/migrate.go` — v9→v10 branch added.
- `internal/db/buglink.go` — new `InsertBuglink(bugKey, runID, reason, linkedBy)` helper. Uses `INSERT OR IGNORE` so the unique constraint silently dedupes hook re-runs.
- `internal/jira/task_done_hook.go` — new file. Defines `BugLinkStore` + `TaskDoneIssueFetcher` interfaces and the `RecordOnTaskDone(fetcher, store, resolver, taskKey)` entrypoint. Logic: load issue → if `IssueType` contains `bug` (case-insensitive — covers `Defect`, `Production Bug`, etc.) → walk `ParseLinkCandidates` (structured "is caused by" links) and `ParseDescriptionTags` (`caused_by:HEX` regex) → resolve each candidate to a runID via the existing `BugLinkResolver` → insert one row per resolved (bug, run) pair.
- `cmd/task.go::runTaskDone` — buglinks hook wired in after `TransitionToDone`. Non-fatal: hook failures are logged but don't block the manual transition. Adds `bugLinkResolverAdapter` (mirrors poller's resolver shape) so cmd/ doesn't need to import jira's unexported poller types.
- `internal/db/qa_queries.go::BugHotspots` — query rewritten. Body now JOINs `buglinks` with `runs` and groups by `COALESCE(NULLIF(r.git_remote, ''), r.cwd)` × `date(r.started_at, 'weekday 0', '-6 days')`, counting `DISTINCT bl.jira_bug_key` per cell. Lint/test deltas no longer feed the count.

### Frontend

- `cmd/web/dashboard/index.html` — Bug Hotspots subtitle updated from "regression proxy" to "distinct Jira bugs whose offending run lives in that (repo, week) — fed by task-done hook". No JS changes — the row/cell shape `{repo, week, count}` is unchanged.

### Demo seed

- `internal/demo/seed.go::SeedCrossProject` — adds 12 `buglinks` rows. Each (project, sprint) pairs one bug to the regressing run (rIdx==3); first sprint of every project pairs a second bug to the same run, exercising the `COUNT(DISTINCT jira_bug_key)` path so the heatmap shows a `2` cell on week `03-30`.
- `ResetDB` also wipes `buglinks` for clean reseeds.

## Notes / debt left

- The hook only fires on `dandori task done <bug-key>`. Bugs closed directly in Jira (no CLI invocation) won't populate the buglinks table — the existing poller `bugLinkCycle` still records `bug.filed` events in that path, but those don't seed buglinks rows. Acceptable for now since the dashboard is fed by the same QA loop that uses the CLI; a later phase can extend the poller to also call `InsertBuglink` when it emits `bug.filed`.
- Bug-type detection uses substring match on `bug` (case-insensitive). False positives are theoretically possible (a custom type named `Bugfix Approved` would match) but harmless — the hook only inserts rows when there's a real "is caused by" link or `caused_by:` tag to resolve.
- The week boundary in the heatmap is computed off the offending run's `started_at` (the moment the regression entered the codebase), not the bug's filed-at. This is deliberate: dashboard users want to see "where did the regression come from", not "when was it noticed". Document this in the user-facing docs in a future cycle.
