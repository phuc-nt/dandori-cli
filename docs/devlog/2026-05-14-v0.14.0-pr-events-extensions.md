# 2026-05-14 — v0.14.0 — pr_events extensions (size, multi-repo, intervention RCA)

## What shipped

Three additive features on top of v0.13's `pr_events` pipeline. No new external integrations; same `dandori sync --github-only` data path.

1. **PR size capture** — `additions`/`deletions` populated from GitHub's `/pulls/{n}` detail endpoint (list endpoint omits them). Surfaces `median_lines_changed` on `/api/metrics/pr-cycle-time` and the dashboard PR-Cycle tile (`· ~N LoC/PR` next to the coverage line).
2. **Multi-repo breakdown** — `?repo=<owner/name>` filter on Trust + PR-Cycle endpoints, new `/api/metrics/repos` lister, dashboard dropdown above the KPI strip when ≥ 2 active repos.
3. **Intervention RCA buckets** — three new `RunOutcomeReason` constants (`wrong_approach`, `scope_misunderstanding`, `missing_context`) emittable via `dandori event --reason`. Closes framework §9 ❌ row.

Total effort came in around the planned 5–7h band. Phases independent except Phase 02 surfacing the Phase 01 size column.

## Architecture

**Schema v13 — idempotent additive ALTER.** Two new nullable INTEGER columns on `pr_events`. SQLite lacks `ADD COLUMN IF NOT EXISTS`, so the migration is a Go function that probes `pragma_table_info` per column before issuing the ALTER. Required because old test fixtures (e.g. `migration_v9_test.go`) reset `schema_version` and re-`Migrate()` against a `SchemaSQL` that already has the v13 columns — without the probe, the v12→v13 step would crash on second run.

**`*int` for nullable counts.** Distinguishes "detail not fetched" (NULL) from "no-op PR" (0). UPSERT uses `additions = COALESCE(excluded.additions, pr_events.additions)` so a later sync without `--fetch-detail` doesn't wipe a previously populated row.

**Two-literal SQL pattern for `?repo=`.** Rather than dynamic SQL concat, both `queryAICFR` and `GetPRReviewCycleTimeByRepo` keep two literal queries (with/without `AND repo = ?`) and pick one based on the arg. KISS, prepared-stmt-friendly, no string-builder injection surface. Combined with regex validation `^[A-Za-z0-9._-]+/[A-Za-z0-9._-]+

 at the endpoint boundary, the parameter never touches a non-bound code path.

**Partial Trust per-repo (`repo_scope: "cfr_only"`).** `task_attribution` (Acceptance) and `runs` (Intervention) have no `repo` column — they're keyed by `jira_issue_key` / `run_id`. Rather than refuse the filter or invent a fake scope, the result honestly carries `RepoScope = "cfr_only"` when a repo is supplied; the dashboard tooltip explains that only AI-CFR scopes. Adding a `repo` column to those tables is deferred (backfill cost) and the partial signal is already actionable.

**CustomEvent for widget coordination.** `<select id="repo-filter">` change → `document.dispatchEvent('dandori:repo-change', {detail:{repo}})`. Trust + PR-Cycle widgets `addEventListener` and refetch. Scope persists via `window.__dandoriRepoFilter` so late-bound widgets (different persona tabs) can read the current scope on initial render. KISS shared state — no Redux, no observable, just a global + an event.

**Intervention buckets via existing event aggregator, no schema change.** The plan's insight: `task_attribution.session_outcomes` is opaque JSON (`map[reason]int`); adding keys is purely application-level. `dandori event --reason wrong_approach` writes `event_type = intervention.wrong_approach`; `ClassifyRunOutcome` recognizes the new event types alongside `test.fail`/`lint.fail`/etc.; the existing `aggregateSessionStats` collects them into the JSON map automatically. Zero migration, zero downstream change to the RCA endpoint.

## Decisions worth remembering

- **NULL vs sentinel for size columns.** `*int` chosen over `int` with `-1` sentinel. Saves a layer of "is this real or a marker" branching everywhere downstream.
- **Repo regex on the boundary, not deeper.** Validation lives in `repos_endpoint.go` + the trust/pr-cycle endpoint handlers — not in the DB layer. The DB sees only parameterised input. If a future caller bypasses the HTTP boundary (e.g. a CLI command), it must validate too.
- **`scope` and `missing` keyword scan in `LegacyBucketReworkReason` accepted as fuzzy.** A free-text "out of scope rejection" now buckets to `scope_misunderstanding` instead of `human_reject`. The intervention bucket is the more specific signal, so the precedence is intentional.
- **Hardcoded label map in `qa.js`** instead of fetching from an endpoint. <15 entries, changes rarely, KISS. If it grows past 20 or starts drifting from `db.ReasonLabel`, expose `/api/meta/reason-labels`.
