# 2026-05-07 — v0.11.2 release (dashboard top-5 metric UX)

## What this tag bundles

v0.11.2 surfaces the 5 solo-engineer metrics that v0.11.0 shipped but buried. Pure frontend patch — no Go changes, no schema, no new endpoints.

The post-v0.11.1 inventory found 82 solo-applicable metrics. The top-5 most useful for self-evaluation (per the audit) were all live behind sub-tabs, tooltip text, or 12px subtitles. This release brings them to the top.

## What shipped

| Fix | Where | What changed |
|-----|-------|--------------|
| **Solo KPI hero strip** | Engineering view top | New 3-tile section: Success Rate / Rework Rate / Avg Cost/Run, each with WoW delta badge + 8-week SVG sparkline |
| **Slope label elevation** | Engineering → trend chart | Slope text moved from 12px muted card-subtitle to 14px accent-colored span next to title |
| **RCA Recent Changes list** | QA → rework doughnut | Top-3 causes ranked by WoW delta with ↑/↓ badges + proportional bars, rendered inline below the doughnut |

## Why this patch

Audit verdict (`plans/reports/explore-260507-1240-dashboard-top5-ux-audit.md`):

> "All 5 metrics are computed and accurate. The problem is UX hierarchy and discoverability, not data. A few surgical fixes to presentation will make these metrics discoverable and actionable within 30 seconds of login."

Concrete pain points the patch addresses:
- Trend metrics (success / rework / cost) all bundled into one multi-series chart on a sub-tab requiring 2+ clicks → now the 7d snapshots are above-the-fold on Engineering landing.
- RCA WoW delta hidden in doughnut tooltip → now an always-visible list with arrow direction + magnitude bar.
- Slope insight ("improving 3.2pp/week") rendered in 12px muted text → now 14px accent-colored next to title.

## Architectural choices

- **Inline-SVG sparklines, not Chart.js.** The existing `.hero-tile` pattern uses Chart.js canvases. Reusing it would have created Chart.js registry contention with the existing trend chart. New `.solo-kpi-tile` class uses inline `<svg>` + a small `buildSparklinePath()` helper. ~40 lines of SVG path math vs another Chart.js instance.
- **No new endpoints.** Hero strip queries the same 3 trend endpoints the existing trend chart uses (`/api/trends/{success-rate,cost,rework-rate}`). RCA list reuses the doughnut's existing `/api/rca/breakdown` fetch.
- **Direction-aware coloring.** Each KPI knows whether "going up" is good or bad. Success rate up = green; rework rate up = red; cost up = red. Encoded in widget config (`higherIsBetter: true|false`).
- **Empty state per tile**, not page-level. If success-rate has data but cost doesn't, only the cost tile shows "No data yet" — the other two still render.

## Drift / correction

Initial implementation hit a wrong endpoint name (`/api/trends/cost-per-run`) per the planning audit. Smoke test against the actual server confirmed the endpoint is `/api/trends/cost` (matches the existing trend chart). Fixed in a follow-up commit (`39ea509`) before tag.

## Lessons

- Audits between releases earn their keep. The top-5 audit + UX audit + walkthrough audit produced 3 ship-ready releases (v0.11.0, v0.11.1, v0.11.2) in 36 hours.
- "Compute and bury" is a recurring failure mode: the slope label was correctly computed in v0.11.0 but rendered too small to read. Computation-correct ≠ user-visible.
- One persona view ("Engineering") doubles cleanly as the solo home. Adding a separate "Solo" persona would have been overkill — the audit explicitly recommended against it, and the hero strip on Engineering proves the point.
